## 1. Objetivo

Capturar, documentar y compartir las lecciones aprendidas durante la ejecucion del proyecto OCR Operativo, con el proposito de:

- Identificar practicas exitosas que deben replicarse en futuros proyectos.
- Documentar problemas encontrados y las soluciones aplicadas.
- Mejorar continuamente los procesos de desarrollo y gestion de proyectos.
- Crear un acervo de conocimiento organizacional.

---

## 2. Metodologia de Captura

### 2.1 Cuando Capturar

| Momento | Descripcion |
|---|---|
| **Continuo** | Cualquier miembro del equipo puede registrar una leccion en cualquier momento durante el proyecto. |
| **Fin de fase** | Al completar cada fase semanal se realiza una revision breve de lecciones. |
| **Retrospectiva final** | Sesion formal de retrospectiva al cierre del proyecto (Dia 28). |
| **Post-incidente** | Despues de cualquier incidente, bloqueo o problema significativo. |

### 2.2 Quien Puede Capturar

Todos los miembros del equipo del proyecto estan invitados y se espera que contribuyan:

- Project Manager
- ML Engineer
- Backend Engineers (1 y 2)
- Frontend Developer
- QA Engineer
- DevOps Engineer
- Stakeholders y usuarios finales (Equipo de Operaciones)

### 2.3 Como Capturar

1. Completar el formato de la seccion 4 (una fila por leccion).
2. Entregar al PM o registrar directamente en este documento.
3. El PM revisa, clasifica y consolida semanalmente.

---

## 3. Categorias de Lecciones

| Categoria | Codigo | Descripcion | Ejemplos |
|---|---|---|---|
| **Proceso** | PROC | Lecciones relacionadas con la gestion del proyecto, metodologia, planificacion y seguimiento | Estimaciones, reuniones, flujos de trabajo, control de cambios |
| **Tecnico** | TECH | Lecciones sobre tecnologia, arquitectura, desarrollo, infraestructura y herramientas | Rendimiento OCR, integracion SAP, configuracion K8s, bugs criticos |
| **Comunicacion** | COMM | Lecciones sobre comunicacion interna, con stakeholders, documentacion y reportes | Malentendidos, alineacion de expectativas, reportes de avance |
| **Riesgos** | RISK | Lecciones sobre identificacion, evaluacion y respuesta a riesgos | Riesgos no previstos, mitigaciones efectivas, contingencias activadas |

## 4. Formato de Registro

Cada leccion aprendida debe documentarse con la siguiente estructura:

### Campos del Registro

| Campo | Descripcion |
|---|---|
| **ID** | Identificador unico (LA-XXX) |
| **Fecha** | Fecha en que se identifico la leccion |
| **Fase** | Fase del proyecto (1, 2, 3 o 4) |
| **Categoria** | PROC / TECH / COMM / RISK |
| **Reportado por** | Nombre y rol del miembro del equipo |
| **Que paso** | Descripcion objetiva de la situacion o evento |
| **Impacto** | Como afecto al proyecto (positivo o negativo) |
| **Que se aprendio** | La leccion o conocimiento derivado de la experiencia |
| **Recomendacion** | Accion concreta para futuros proyectos |
| **Tipo** | Exito (replicar) / Problema (evitar) / Mejora (optimizar) |

## 6. Guia de Retrospectiva Final

La retrospectiva final del proyecto se realizara el **Dia 28** con todo el equipo. A continuacion la guia para facilitar la sesion.

### 6.1 Agenda de la Retrospectiva

| # | Actividad | Duracion | Facilitador |
|---|---|---|---|
| 1 | Apertura y contexto | 10 min | PM |
| 2 | Revision de metricas del proyecto | 15 min | PM |
| 3 | Que funciono bien (replicar) | 20 min | Todo el equipo |
| 4 | Que no funciono bien (evitar) | 20 min | Todo el equipo |
| 5 | Que podemos mejorar (optimizar) | 15 min | Todo el equipo |
| 6 | Consolidacion y priorizacion | 10 min | PM |
| 7 | Cierre y proximos pasos | 5 min | PM |
| | **Total** | **1.5 horas** | |

### 6.2 Preguntas Guia por Categoria

#### Proceso (PROC)
- Las estimaciones de tiempo fueron realistas?
- El cronograma de 4 semanas fue adecuado?
- El proceso de control de cambios fue util?
- Las reuniones diarias aportaron valor?
- La planificacion por fases fue efectiva?

#### Tecnico (TECH)
- La arquitectura elegida fue la correcta?
- Google Cloud Vision cumplio las expectativas de precision?
- La integracion con SAP presento desafios inesperados?
- El stack tecnologico fue adecuado?
- La infraestructura K8s/GCP fue estable?
- Hubo deuda tecnica significativa?

#### Comunicacion (COMM)
- La comunicacion con el equipo de operaciones fue efectiva?
- Los reportes de avance fueron claros y oportunos?
- Hubo malentendidos que afectaron el proyecto?
- La documentacion fue suficiente?
- Los canales de comunicacion fueron los adecuados?

#### Riesgos (RISK)
- Se identificaron los riesgos correctos al inicio?
- Hubo riesgos que no se anticiparon?
- Las estrategias de mitigacion fueron efectivas?
- El presupuesto de contingencia fue suficiente?
- Como se manejaron los problemas inesperados?
